# Data Layering and Versioning

This repository uses layered data directories to make lineage explicit and reproducible.

## Layers

- `data/raw/`: immutable original downloads or exports.
- `data/external/`: external side inputs not generated by this project.
- `data/interim/`: cleaned but still row-oriented working tables.
- `data/processed/`: model-ready canonical datasets.
- `data/features/`: feature store style artifacts (Parquet tables, index mappings).
- `data/reference/`: static lookup tables and metadata.

## Lineage Rule

Every material dataset in `interim/`, `processed/`, or `features/` should have:

1. A reproducible build command.
2. An upstream source list.
3. A version manifest under `configs/data/versions/`.

## Versioning Command (DVC-equivalent)

Use `src.data.versioning` to snapshot file hash, schema summary, and lineage metadata:

```bash
scripts/run_in_pytorch_env.sh python -m src.data.versioning \
  --dataset-id traverse_city_clean_v1 \
  --layer interim \
  --target-path data/interim/traverse_city_daytime_clean_v1.csv \
  --source-path data/processed/tabular/traverse_city_daytime_meteo_preprocessed.csv \
  --build-command "python -m src.data.clean --input ... --output ..." \
  --notes "T31 baseline clean dataset"
```

Manifest output:

- `configs/data/versions/traverse_city_clean_v1.json`

## Storage Notes

- Large generated artifacts should stay ignored by git and be reproducible from code.
- Keep human-readable manifests small, deterministic, and committed.
